# Отчёт по лабораторной работе №3  
## Разработка клиент-серверного взаимодействия на React (каталог товаров, DummyJSON Products)

---

## 1. Цель работы

Освоить принципы клиент-серверного взаимодействия в React-приложении, научиться получать и отображать данные из внешнего API, обрабатывать основные состояния сетевого взаимодействия (загрузка, успех, ошибка), а также связать интерфейс лабораторной работы №2 с реальными HTTP-запросами к mock API.

---

## 2. Задание

**Вариант:** каталог товаров (п. 2 методички).

**Требования к выполнению:**

- На основе приложения лабораторной работы №2 реализовать клиент-серверное взаимодействие с использованием внешнего mock API (**DummyJSON Products**).
- Загрузка каталога товаров с сервера, отображение карточек, просмотр товара, поиск по названию.
- Желательно: поиск, фильтрация (в т. ч. по категориям), пагинация, работа с query-параметрами в URL, имитация POST/DELETE там, где API это позволяет.

**Что сделано в проекте (кратко):**

| Элемент | Реализация |
|--------|------------|
| Источник данных | `https://dummyjson.com` (Products, Categories) |
| Список и пагинация | `GET /products`, `GET /products/category/:slug` с параметрами `limit`, `skip` |
| Категории | `GET /products/categories`, вкладки «Все категории» + slug категорий |
| Поиск по русскому/английскому названию | клиентская фильтрация по полному набору товаров (кэш после загрузки), эвристика перевода названий в `src/i18n/ruCatalog.js` |
| URL | синхронизация `q`, `cat`, `page` через `history.replaceState` |
| Карточка товара | `GET /products/:id` |
| Имитация POST/DELETE | `POST /products/add`, `DELETE /products/:id` из карточки товара (демонстрация запросов; данные на сервере не сохраняются постоянно) |

## 4. Пошаговое описание выполнения работы

1. **База от лаб. №2.** Сохранена структура интерфейса: шапка, вкладки категорий, список карточек, корзина, боковая панель, модальное окно товара, пагинация, Tailwind, светло-зелёная тема.

2. **Слой API.** Создан модуль `src/api/dummyJson.js`:
   - функции `fetchCategories`, `fetchProducts`, `fetchProductById`;
   - нормализация ответа API к полям `id`, `name`, `price`, `category`, `imageUrl`, `description` (конвертация цены USD→RUB в `src/config/pricing.js`);
   - для поиска по всему каталогу — постраничная выгрузка товаров в память с кэшем и повторным использованием одного запроса;
   - таймауты HTTP-запросов, чтобы зависший ответ не оставлял экран в бесконечной загрузке;
   - `prefetchCatalogForSearch()` — фоновая подгрузка полного каталога для ускорения поиска;
   - имитация `addProductMock` / `deleteProductMock` для демонстрации POST/DELETE.

3. **Локализация интерфейса названий.** Файл `src/i18n/ruCatalog.js`: словарь категорий и эвристика `localizeProductTitle` для отображения названий на русском; поиск по подстроке в русском и английском варианте названия (`rawProductMatchesTitleQuery`).

4. **Состояние в `App.jsx`.** Через `useReducer` хранятся: статус экрана (`loading` / `list` / `empty` / `viewing` / `error`), список товаров, `total`, `skip`, `limit`, строка поиска, активная категория, выбранный товар, ошибка, флаги загрузки. Отдельно `useState` для корзины и списка категорий.

5. **Синхронизация с URL.** При изменении поиска, категории и страницы обновляется строка запроса (`q`, `cat`, `page`) без перезагрузки страницы; при первом открытии состояние читается из `window.location.search`.

6. **Обработка гонок запросов.** Номер «поколения» запроса (`fetchGenerationRef`): устаревший ответ не перезаписывает состояние, если пользователь уже изменил фильтры или запрос.

7. **Оптимизация поиска.** Фильтрация выполняется по сырым объектам API; тяжёлый `mapProduct` (картинки, поля карточки) вызывается только для текущей страницы результатов, чтобы не блокировать интерфейс на большом каталоге.

8. **Прокси для изображений (dev).** В `vite.config.js` для пути `/cdn-proxy` настроен прокси на CDN DummyJSON, чтобы уменьшить проблемы с загрузкой картинок в режиме разработки.

9. **Сборка и проверка.** Выполняются команды `npm run build` и при необходимости `npm run preview` для проверки production-сборки.

## 5. Описание реализации управления состояниями сетевого взаимодействия

| Состояние | Как проявляется в UI | Как обрабатывается в коде |
|-----------|----------------------|---------------------------|
| **Начальная загрузка** | Текст «Загрузка каталога…», пока список пуст и идёт первый запрос | `status: 'loading'`, `listLoading: true`, `dispatch(FETCH_LIST_START)` перед `fetchProducts` |
| **Загрузка при смене фильтра** | Подсказка «Обновление списка…», при наличии уже показанных карточек | `FETCH_LIST_START` выставляет `listLoading: true`; список не очищается до прихода нового ответа |
| **Успех** | Сетка карточек, пагинация, при необходимости «Нет товаров по заданным условиям» | `FETCH_LIST_SUCCESS`: запись `catalog`, `total`, `skip`, `limit`, `listLoading: false`, переход в `list` или `empty` |
| **Ошибка** | Блок «Ошибка загрузки» и текст сообщения (в т. ч. таймаут) | `FETCH_LIST_ERROR`: очистка каталога, `status: 'error'`, `listLoading: false` |
| **Просмотр товара** | Модальное окно с деталями | `SELECT_PRODUCT` / `CLOSE_PRODUCT`, статус `viewing` |
| **Корзина** | Локальное состояние `cartItems` в `App`, без сервера (как типичный UX для лабораторной модели) | `useState`, добавление/удаление позиций |

Дополнительно: **для карточки товара** в `ProductDetail` при загрузке по `id` используется локальное состояние загрузки/ошибки; при успешном ответе имитации POST/DELETE показывается уведомление с телом ответа API.

## 7. Вывод

В ходе работы приложение каталога товаров из лабораторной работы №2 переведено на **клиент-серверную модель**: данные загружаются с **DummyJSON Products**, категории — с эндпоинта категорий, реализованы **поиск**, **фильтрация по категориям**, **пагинация**, **синхронизация с query-параметрами URL**, отображение **состояний загрузки и ошибки**, **просмотр товара** по `id`, **имитация POST и DELETE** при работе с API. Дополнительно реализованы **локализация названий** для пользовательского интерфейса, **кэширование полного каталога** для поиска и **меры против зависания интерфейса** (таймауты запросов, оптимизация обработки списка при поиске).  

Работа позволила закрепить навыки работы с `fetch`, асинхронной логикой в React, `useReducer`/`useEffect` для связи UI с данными сервера и практическую обработку типичных сценариев сетевого взаимодействия.
